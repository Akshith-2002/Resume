Open Credits

// AI-assisted EdTech platform — vibe-coded build

100%vibe-coded build
2flows · learning · analytics
E2EPM ownership

Role

Project Manager and builder. Wrote the spec, ran AI-assisted full-stack development for the platform, set up analytics pipelines and event-tracking, and shipped to learners. The whole vertical, in one head.

Stack

  • Vibe coding (AI-assisted)
  • SQL
  • n8n
  • Event tracking
  • Analytics pipelines

Context

Can a PM build the whole thing if the spec is sharp?

Open Credits started as a question more than a brief: how much of an EdTech platform can a PM build solo, end-to-end, when the build tool is AI-assisted development and the work is mostly translation between a clear spec and the model that's writing the code?

The answer turned out to be: more than I expected, because the bottleneck stops being syntax and starts being the precision of the thing you're asking for.

Approach

Spec sharpness compounds

Vibe coding doesn't fail because the model can't write code — it writes code instantly. It fails when the spec is fuzzy, because then the code is fluent but wrong. So the work moves upstream: define the data model, the event schema, the user flows, the acceptance criteria, with more precision than you would for human engineers, because there's no implicit common-sense filter to catch ambiguity.

  • Data model first. Learner, course, module, attempt, completion, credit. Five entities, every relationship explicit, before any UI thinking.
  • Event schema second. Every meaningful learner action emits a typed event. The analytics dashboard wasn't a downstream afterthought — the events were the spec.
  • Flow specs third. Onboarding, course discovery, lesson consumption, attempt + grade, credit issuance. Each flow as a state machine with named transitions.
  • Then build. The actual coding was hours, not weeks, because the hard work was already done.

Architecture

Two flows, shared event spine

[ Learner ]
   ↓
[ Learning surface ]  ─── events ───▶ [ Event spine ]
   // course · lesson · attempt            ↓
                                       [ Analytics pipeline ]
                                            ↓
                                       [ Reporting / insights ]
                                            ↑
[ n8n automations ] ── triggers ─────────────┘
   // reminders · grade nudges · cohort comms

The event spine is the architectural decision that paid back. Every action becomes a typed event. Reporting reads from the event log. Automations subscribe to event patterns. Add a new flow later? Define its events, then build the UI. The plumbing already exists.

Vibe coding works when the spec is sharp. The model is faster than you at writing code. It is not faster than you at deciding what the code should do.

Outcome

Shipped — and proved a thesis

The platform shipped with both flows working: the learner side (course consumption, attempts, credits) and the analytics side (engagement, completion, cohort performance). All built solo as PM via AI-assisted development.

Bigger outcome than the build itself: a working answer to "can a sharp PM ship a platform?" — yes, when the constraint stops being typing speed and becomes spec quality. That changes how I scope future engagements.

What I'd do differently

Write a test plan before the spec

The biggest learning: AI-assisted code lands quickly, but verifying it lands correctly is its own problem. If I started this again I'd write the test plan — what acceptance scenarios prove this flow works, what failure modes I want to catch — before I wrote the spec. The model can write tests too, but only as well as the cases I describe.
