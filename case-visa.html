Visa Application Automation

// an agent that applies for visas on government portals — login to document upload

50+countries targeted
7tab wizard automated
0manual steps to submit

Role

Solo builder — architecture, agent design, and delivery. Vibe-coded the harness, designed the country-plugin pattern, and drove it live end-to-end against a real government portal.

Stack

  • Python
  • Playwright
  • Claude Vision
  • Postgres
  • Docker

Context

Visa filing is high-volume, high-tedium, zero-creativity work

A travel operator files the same visa applications thousands of times — same wizard, same fields, same document uploads, a fresh CAPTCHA every session. It's exactly the kind of work a human should never have to do twice, and exactly the kind of work most automation can't touch because the portals fight back.

Government portals are hostile to automation by design: image CAPTCHAs, session popups, AJAX-loaded dropdowns, and fields that reject anything but Arabic script. The goal was an agent that treats one portal as a plugin — get Oman working end-to-end, then replicate the pattern to the next 50+ countries.

Approach

A shared harness, one plugin per country

The architecture separates what's universal from what's country-specific. A shared core handles the browser, CAPTCHA solving, OCR, config, and the database. Each country is a plugin that only encodes its own portal flow and field map. Oman was the reference implementation.

  • CAPTCHA: Claude vision reads the image challenge. A digits-only heuristic plus a 4× image upscale pushed accuracy to near-100%; refusals are handled with a refresh-and-retry.
  • OCR & profile: Claude vision OCRs the applicant's passport and residence card into a structured profile — including the Arabic name, place of birth, and place of issue that three portal fields require and reject Latin text for.
  • Data store: profiles land in Postgres (Dockerised) and a per-applicant JSON the apply flow drives from.
  • Navigation: the Liferay portal aborts on direct URLs, so the agent navigates by clicks only — through the eligibility wizard, the registered-user popup, and the second login into the application form.

Architecture

Documents in, filed application out

[ Passport + residence card ]
      ↓     Claude vision OCR → structured profile (incl. Arabic fields)
[ Postgres + customer.json ]
      ↓
[ Playwright agent ] ── login ──▶ CAPTCHA solved by Claude vision
      ↓
[ 7-tab wizard ]  travel docs → additional details → document upload
      ↓
[ STOP before submit ]  // human reviews + clicks Submit

The 7-tab wizard fills automatically through the required-documents upload: 14 fields on the travel-document tab, business details on the next, then four document uploads (each behind a hidden file input and a modal that has to be dismissed between uploads). Then it stops.

The agent deliberately never clicks Submit. It fills everything, leaves the browser open on the review screen, and hands the final decision to a human. Automating the tedium is the win; automating accountability is not.

Outcome

Oman filed end-to-end, hands-off until review

For the Oman portal, an applicant's documents go in and a fully-filled application comes out — login, CAPTCHA, eligibility, all seven tabs, and every required document uploaded — with no human keystrokes until the review-and-submit step. The country-plugin pattern means the next portals reuse the entire core; only the flow map changes.

What I'd do differently

Treat portal fragility as a first-class feature

Government sites throw transient server errors and change markup without notice. The next iteration leans harder on resilience — checkpointing progress per tab, resuming mid-application, and alerting a human the moment a portal genuinely breaks rather than retrying into a wall.
