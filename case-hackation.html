Context

The brief: build an agent that does something useful in EV mobility

The HackAtion 2025 AI Agents track asked builders to ship something that could replace a workflow a human currently has to slog through. Most teams went straight to dashboards-with-an-agent-attached. I went the other direction: no UI at all.

EV charging in India today is a coordination tax on the driver: check an app for stations, check another for availability, drive there, find it occupied, drive again. It's the kind of friction that any human would rather hand to someone else if they could.

Approach

The agent should do the boring part

Three design calls, made early:

  • Telegram over a custom UI. Zero install. The user is already there. The interaction is a chat, not a screen, which matches what an agent actually does.
  • Three capabilities, not ten. Slot booking, payment handling, live availability. Pick the spine of the workflow, ship those, leave the rest. Hackathon scope discipline.
  • n8n for orchestration, GPT for conversation. n8n handles the booking state machine, payment APIs, and availability polling. GPT is the front-of-house. Each tool does what it's good at.

Architecture

How the agent moves

[ User on Telegram ]
       ↓     "Find me a charger near Indiranagar at 7pm"
[ Telegram Bot API ]
       ↓     webhook
[ n8n orchestration ]
       ├─▶ [ GPT ]            // intent + slot extraction
       ├─▶ [ Availability API ] // live station status
       ├─▶ [ Booking engine ]   // reserve slot
       └─▶ [ Payment gateway ]  // confirm + charge
       ↓
[ User gets ] ── confirmed slot, location pin, payment receipt

The interesting part: GPT only handles natural language, never makes the booking decision. n8n holds state and gates every action. That separation means the agent can't hallucinate a reservation — it can only describe what n8n actually did.

Agents win when they handle the orchestration users would otherwise abandon. No one's switching from "open three apps in sequence" to "open three apps, plus a chat assistant." They're switching to "tell the bot what you need."

Outcome

Won the AI Agents track

The judges singled out the agent / orchestration / tool boundary — the thing that made the system feel reliable rather than like another hallucination demo. The Telegram-first surface was the second most-asked question in the Q&A: most teams had built a dashboard, and the demo that didn't need one stood out.

Build window: about 36 hours, mostly solo. The work that mattered most wasn't the code — it was deciding what NOT to build before the clock started.

What I'd do differently

Add memory earlier

The hackathon entry was stateless per-session. To turn this into a real product I'd give the agent persistent memory of a user's home and work addresses, preferred station types (DC fast vs AC), and historical availability patterns. Then "find me a charger" becomes "the agent already knows" — which is the actual win.
