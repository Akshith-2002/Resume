Role

Architect of the automation and the data integration. Wrote the data contracts that unblocked cross-channel ROAS, ran the n8n workflow design, and partnered with engineering on the Power BI delivery.

Stack

  • BigQuery

  • Power BI

  • Meta Ads API

Context

Two problems wearing the same coat

A growing tours operator with two pain points that looked separate but shared a root cause: visa processing was bottlenecked by humans who'd already become a process, and marketing decisions were being made on numbers no one could fully trust.

The visa team was hand-stitching applications across channels. The growth team had three ad platforms, four spreadsheets, and a working theory of ROAS. Both problems had the same shape: data trapped in places it couldn't flow out of, and decisions waiting on humans to copy-paste.

Approach

Make the data contract the deliverable

I treated the marketing-data work and the visa-ops work as one system with two surfaces. The first artefact wasn't a dashboard — it was a data contract spec defining: every event we'd capture, what dimensions it carried, where it landed in BigQuery, and which downstream consumer relied on it. Once that document existed, parallel build streams stopped colliding.

  • Visa automation: n8n orchestrates intake → document validation → status updates → applicant comms. Capacity scales to up to 4,000 applications a day without touching the headcount.
  • Marketing data unification: Google Ads + Meta Ads + GTM events all ingest into BigQuery on a shared schema. Every conversion has a source-of-truth row.
  • Power BI delivery: ROAS, CAC, channel-mix, funnel stage — all live, all reconcilable to source. No more "which number is right" stand-up debates.
  • A/B testing pipeline: hypotheses defined upfront, GTM events instrumented with engineering, results reported for go/no-go decisions.

Architecture

The data flow

[ Google Ads ]    [ Meta Ads ]    [ GTM events ]
       ↘            ↓↙
            [ BigQuery (source-of-truth) ]
                       ↓
            // data contract: shared schema, one ROAS
                       ↓
              [ Power BI dashboards ]

[ Visa applicant ] ──▶ [ n8n intake ]
                            ↓
                       [ doc validation ]
                            ↓
                       [ status engine ]  ──▶ [ applicant comms ]
                            ↓
                       [ BigQuery audit ]

The shared design choice across both halves: BigQuery as the source-of-truth. Marketing rolls up to it for reporting; visa operations roll up to it for audit and capacity planning. One system to instrument, one place to query, one set of conventions to learn.

The contract is the asset, not the dashboard. Once Google + Meta + GTM spoke the same schema, the dashboard was a few hours of work — the months of value lived in everything you stop arguing about.

Outcome

Capacity unlocked, time recovered, decisions sharper

Visa processing scaled to support up to 4,000 applications a day with no proportional headcount increase. Marketing reporting that previously consumed ~8 hours a week of manual reconciliation now runs continuously. The growth team makes channel-allocation decisions on numbers that match across systems, which is the boring foundation under every "data-driven" claim.

The work is ongoing. New ad surfaces (TikTok, search alternatives) plug into the same schema as they're added. The visa workflow keeps absorbing edge cases without rewrites because the contract was designed to be additive.

What I'd do differently

Start with attribution decisions, not attribution data

The data was the easy part. The harder part — and the one I'd timebox sooner next time — was getting stakeholder agreement on attribution windows, channel-credit splits, and how to reconcile Meta's view of a conversion with Google's. Doing the data work first and the policy work second means a moment, late in the project, where everyone realises the numbers are correct but no one likes the answer.
