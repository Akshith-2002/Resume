StackD Smart Vending

// AI + IoT operations platform · led 9-person cross-functional team

9person team led
~30%stock-outs reduced
3layers · HW · BE · BI

Role

Business Analyst & Project Manager. Owned discovery, requirements, technical specs, sprint delivery, UAT, and launch. Acted as the client-facing PM and the bridge between hardware, backend, and BI engineers.

Stack

  • Python (AI)
  • n8n
  • Tableau
  • IoT sensors
  • JIRA
  • SQL

Context

Vending economics fail quietly. We made the failure loud.

Smart-vending operators lose money in two places at once: empty machines that can't sell, and full machines no one's buying from. Both look identical from a back-office spreadsheet — until you've walked the route.

The brief from the client was deceptively simple: build a platform that tells operators what to restock, where, and when. The hard part was that the answer lives across three data planes that don't talk to each other — IoT sensors on the machine, transaction data in the backend, and field-ops decisions made from gut feel.

Approach

Spec the boundary, not the feature

My first artefact wasn't a wireframe or a sprint plan. It was a one-page data contract: for every event the machine emits, what does it mean, who owns it, and which downstream system depends on it? That contract became the acceptance criteria for hardware, the schema for the backend, and the dimensions the BI layer would aggregate against.

  • Discovery: field-walked four routes with operators. Watched the actual restock decision get made. The spec changed.
  • BRDs & FRDs: three documents — one per data plane (IoT, backend, BI) — with explicit handoff specs between them.
  • Sprint cadence: two-week sprints in JIRA. Hardware shipped first because everything else was downstream of clean signal.
  • UAT: staged on a single high-traffic machine before any rollout decisions were made.

Architecture

The data flow we shipped

[ IoT sensors ]
      ↓     telemetry · weight · door-events · vends
[ Backend ingestion ]
      ↓     Python AI: predictive restock model
[ Decision engine ] ── n8n workflow ──▶ [ Field ops alerts ]
      ↓
[ Tableau BI layer ]   // remote monitoring + anomaly detection

Predictive over threshold. The naive version of restock alerts is a level threshold ("alert when stock < 30%"). That's right twice a day and wrong the rest of the time. The Python model predicts depletion based on machine-specific velocity, day-of-week, and weather signal — so the alert fires when a route operator can still get there before the machine empties, not after.

Anomaly detection earned its place the day a machine reported normal vends while its actual stock was unchanged. The model flagged it before the operators did. Revenue saved that month paid for a quarter of the build.

Outcome

~30% fewer stock-outs. The number that matters.

Across the rollout cohort, stock-out incidents dropped roughly 30% in the first quarter post-launch. That's the metric the client signed for. The ones we didn't put in the SOW: anomaly-detection caught two hardware malfunctions that would otherwise have been invisible for weeks, and the Tableau dashboard cut the operator's weekly review from a half-day to about twenty minutes.

The 9-person team — engineering, design, data, ops — ran to the cadence I set in JIRA from sprint one through launch. No fire drills. No "where are we" Slack threads on Sunday. That's the part of PM work that doesn't fit on a slide but is the actual job.

What I'd do differently

Ship the BI layer earlier

We sequenced hardware → backend → BI. It's the right dependency order, but it meant operators didn't get a dashboard until late in the build. If I ran this again I'd ship a stub Tableau view in week three using mocked data, so stakeholders had something to react to before the real signal flowed in. Earlier reactions = earlier course corrections.
